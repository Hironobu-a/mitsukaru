## エグゼクティブサマリー

ミツプロ（https://tax.mitsukaru-pro.co.jp/）は**税理士業界特化型転職・求人プラットフォーム**です。

### サービス概要
税理士・税務担当者と税理士事務所をマッチングする両面型プラットフォームとして、以下の機能を提供：
- **求職者向け**: 求人検索・応募、転職ノウハウ、年収診断、ブラック事務所診断、スキル診断
- **採用企業向け**: 求人掲載、応募者管理（審査制によりホワイト求人に限定）
- **コンテンツ**: SEO対策を意識した業界情報・転職ガイド等

### ビジネスの位置づけ
- ミツカル本体（mitsukaru-corp.co.jp）: 企業向けコンサルティング・M&A支援
- **ミツプロ（tax.mitsukaru-pro.co.jp）**: 税理士業界特化の人材マッチングプラットフォーム

→ ドメイン分離により、B2B（企業）とB2C（個人求職者）のUXを最適化

### 戦略的価値
単なる求人サイトではなく、**専門職マーケットプレイス → データ資産化 → 業界エコシステム構築**という中長期戦略：
1. 税理士労働市場データの蓄積
2. 候補者スキルプロフィールのデータベース化
3. 将来的なコンサルティング支援・教育コンテンツへの展開可能性

**現状の最優先課題（Tech Debt）**:
詳細は `TECH_DEBT_ROADMAP.md` を参照してください。
1. **セキュリティ**: 過去に一部認証情報の追跡履歴あり（ローテーション必須）。
2. **基盤老朽化**: Laravel 8 (PHP 8.0) のサポート終了対応。
3. **技術刷新**: フロントエンド技術の統一化。

**今後の提案方針**:
**「安定運用の確保」と「段階的な技術刷新」**を並行して進めることを提案します。
- **全エンジニア**: 開発・実装を分担して進める
- **正社員エンジニア**: 加えて技術選定サポート・インフラ整備・コードレビューを担当
- **業務委託エンジニア**: テスト拡充にも注力

### 1.2 ビジネスドメインの理解

本サービスは以下のようなユーザー体験を提供しています：

**求職者（税理士・税務担当者）向け**:
- **求人検索**: 条件絞り込み、新着・おすすめ求人の閲覧
- **会員機能**: 無料会員登録、お気に入り、応募管理
- **診断ツール**: 年収診断、スキル診断、ブラック事務所診断
- **情報コンテンツ**: 転職ノウハウ、業界動向、キャリアガイド

**採用企業（税理士事務所）向け**:
- **求人掲載**: 審査制（ホワイト求人に限定）による質の担保
- **応募者管理**: マッチング、選考進捗管理
- **採用支援**: 求人原稿作成サポート、採用ノウハウ提供

**データ戦略**:
- 専門職の労働市場データ蓄積
- ユーザー行動ログによるマッチング精度向上
- スキルデータベース化による将来的な教育・コンサル展開

### 1.3 技術ビジョン（3年後）

- **安定性**: セキュリティパッチが継続的に提供される最新Laravelを利用
- **開発効率**: フロントエンドを1〜2種類の技術に統一し、新機能開発を加速
- **品質**: テストカバレッジ70%以上、CI/CDによる自動品質チェック
- **拡張性**: APIファーストな設計により、モバイルアプリや外部連携に対応可能
- **データ活用**: 求職者・求人データの分析基盤構築、レコメンド機能の高度化

### 1.4 UX設計マッピング（求職者側ジャーニー）

本サービスは単なる求人検索サイトではなく、**「診断→関係構築→人材DB化」まで設計されている構造**です。

#### レイヤー0: 流入設計（SEO / 広告）

入口は3種類に分かれます：

- **SEO流入**: `税理士 転職` / `税務 年収` などの検索クエリ
- **診断コンテンツ**: ブラック事務所診断などで興味喚起
- **スカウト・広告流入**: 直接的なリーチ

**設計意図**:  
求人検索を主軸にせず**診断コンテンツを入口**にしている点が重要です。これは**データ取得効率を最大化するための戦略**です。

#### レイヤー1: 求人探索

**機能**:
- 条件検索（エリア / 年収 / 事務所規模などのフィルタ）
- 求人詳細閲覧
- お気に入り保存

**UX上の意図**:
1. まず市場の相場感を理解させる
2. 自分の市場価値を比較させる
3. 「気になる」を蓄積させ、後段の登録導線に誘導

#### レイヤー2: 診断・コンテンツ（コア戦略）

**機能**:
- 年収診断
- ブラック事務所診断
- スキル診断
- 転職ノウハウ記事

**実質的な役割**: 自己開示フォーム

ユーザーは診断を受けるために以下を入力します：
- 経験年数
- 資格有無
- 現在の年収
- 志向性（ワークライフバランス、年収重視など）

**この時点で、かなり高精度な候補者データが取得されます。**  
→ **このレイヤーがミツプロのコア価値**です。

#### レイヤー3: 会員化（マイページ）

**機能**:
- 会員登録
- プロフィール保存
- お気に入り管理
- 診断結果保持

**UX意図**:  
「転職するか未定でも、とりあえず登録しておく」状態を作る。  
ここから**タレントプール（人材候補者データベース）**が形成されます。

#### レイヤー4: エージェント接続

**機能**:
- キャリアアドバイザー面談
- 非公開求人紹介
- スカウト

**ここで初めて人が介在**します。

**全体の構造**:
```
SEO流入
  ↓
診断で情報取得（自動）
  ↓
会員化（データ蓄積）
  ↓
データ分析（自動）
  ↓
人的マッチング（人力）
```

これは**半自動×人力ハイブリッドモデル**です。

### 1.5 データ構造的な本質

このサイトは実質、**「税理士業界の人材データベース構築装置」**です。

**蓄積されるデータ**:
- 資格ステータス（税理士 / 科目合格 / 実務経験者）
- 実務領域（相続 / 法人税務 / 資産税 / 国際税務）
- 年収レンジ
- 転職意欲レベル
- 勤務地希望
- 事務所評価（診断結果から）

**将来的な活用可能性**:
- **報酬相場レポート**: 「税理士の平均年収は○○万円」などの市場調査データ
- **市場動向分析**: 人材需給バランス、地域別傾向
- **M&A支援との連携**: 税理士事務所の経営者向けサービスへの展開
- **教育・コンサル**: スキルギャップを補うための研修サービス

**ここが単なる求人サイトとの決定的違いです。**

### 1.7 組織戦略としての読み解き

これは**「国家資格特化型垂直統合モデル」**です。

**拡張ステップ**:
```
1. 税理士を集める（転職サイト）
   ↓
2. 税理士事務所と接点を持つ（採用支援）
   ↓
3. 事務所の経営課題を把握
   ↓
4. M&Aやコンサルへ拡張（corp事業）
```

**人材紹介は"入口商品"の可能性があります。**

もしこれが本気戦略なら、将来的には以下まで伸びます：
- 税理士事務所M&A
- 士業DX支援
- バックオフィス統合（採用・経理・ITインフラ）

### 1.8 戦略的問い: このサービスの本質は何か

重要な問いがあります：

この構造は、

1. **「転職エージェント」なのか**
2. **「士業業界OS構築」なのか**

もし後者なら、UXはもっと以下の方向へ進化する必要があります：

- **継続利用前提**: 転職後も使い続けるプラットフォーム
- **コミュニティ化**: 税理士同士の情報交換・知見共有
- **スキル可視化**: 継続的なスキルアップ・資格管理機能

**技術戦略はこの方向性によって大きく変わります。**  
単なる求人サイトなら、Laravel + MySQL で十分ですが、  
業界OSを目指すなら、API基盤・リアルタイム通信・分析基盤が必要になります。

### 1.9 ビジネス価値創出の視点

現在の資産（UX設計、データ構造、ドメイン体系）から、**マーケッター視点**と**CTO視点**でどのような価値を生み出せるかを整理します。

#### A. マーケッター視点での価値創出

##### 1. データドリブンマーケティングの高度化

**現状の資産**:
- 診断データ（年収、経験年数、資格、志向性）
- 求人閲覧・お気に入り行動ログ
- 応募・面談データ

**生み出せる価値**:
```
入力データ              → 分析        → アウトプット
─────────────────────────────────────────────────────
年収×経験年数×エリア   → セグメント分析 → 超精密ペルソナ
診断結果×求人閲覧     → 相関分析     → 需要予測モデル
流入元×コンバージョン  → チャネル分析  → 獲得単価(CPA)最適化
```

**具体例**:
- **ペルソナ精度向上**: 「30代前半、法人税務5年、年収450万、ワークライフバランス重視」といった超詳細セグメントでの広告配信
- **離脱ポイント特定**: 「年収診断→会員登録」の離脱率が40%なら、診断結果の見せ方を改善
- **コンテンツ最適化**: 「ブラック診断からの流入が最もCVR高い」→ このコンテンツに予算集中

**ROI**: 広告費の無駄打ちを30-40%削減可能

##### 2. オリジナルコンテンツによるSEO支配

**現状の資産**:
- 蓄積された年収データ
- 地域別・職種別の求人動向
- ユーザー行動から見える「悩み」のパターン

**生み出せる価値**:

| コンテンツ種別 | データソース | SEO効果 | 被リンク効果 |
|--------------|------------|---------|------------|
| **税理士年収白書2026** | 診断データの集計 | 「税理士 年収」でTOP3 | メディア引用で自然リンク獲得 |
| **ブラック事務所ランキング** | 診断結果×応募データ | 「ブラック 税理士事務所」独占 | Twitter/noteで拡散 |
| **地域別求人倍率レポート** | 求人数×会員数 | 「○○県 税理士 求人」で上位 | 地方メディアに露出 |
| **転職成功パターン分析** | 応募→内定データ | 「税理士 転職 成功」で上位 | キャリア系メディアから引用 |

**戦略的価値**:  
競合が真似できない**一次データ保有者**としてのポジション確立。  
→ 「税理士の転職情報 = ミツプロ」というブランド刷り込み。

##### 3. リターゲティング精度の革命的向上

**従来の求人サイト**:
- 「サイト訪問者」という粗いセグメントでリターゲティング
- 年収も経験も資格もわからないまま広告配信

**ミツプロの場合**:
```
診断完了ユーザーには以下が判明：
├── 税理士資格: 有/無
├── 経験年数: 3年 / 5年 / 10年+
├── 現年収: 300万台 / 400万台 / 500万+
├── 志向性: 年収重視 / ワークライフバランス重視
└── 転職意欲: 顕在層 / 潜在層
```

**広告メッセージの最適化例**:
- 「資格あり×年収400万×転職顕在」→ 「年収600万以上の求人、今なら○件」
- 「資格なし×経験3年×潜在層」→ 「科目合格者歓迎の事務所特集」
- 「資格あり×ワークライフバランス重視」→ 「残業月20時間以内の求人」

**効果**: クリック率(CTR)が従来の2-3倍、コンバージョン率(CVR)が1.5-2倍

##### 4. ブランディング戦略: 業界標準としての地位確立

**戦略の骨子**:  
人材紹介会社ではなく、**業界インフラ提供者**として認識させる。

**具体施策**:
```
Phase 1: データ公開（信頼獲得）
└→ 年収白書、市場動向レポートを無料公開
   └→ メディア引用、業界団体での引用

Phase 2: 権威性確立（専門家ポジション）
└→ 「税理士転職市場の専門家」としてメディア出演
   └→ 日経・東洋経済等への寄稿

Phase 3: エコシステム形成（囲い込み）
└→ 税理士事務所向けの無料レポート提供
   └→ 「採用する側」も依存する状態へ
```

**長期的価値**: 広告費をかけずとも「税理士 転職」「税理士 年収」で自然に検索される状態。

##### 5. コミュニティマーケティングへの発展

**現在**: 一方向の求人情報提供  
**将来**: 双方向のコミュニティプラットフォーム

**実現すべきUX**:
- 税理士同士の情報交換（事務所の評判、業務ノウハウ）
- 勉強会・セミナー情報の共有
- メンター制度（ベテラン税理士×若手）

**マーケティング効果**:
- **ユーザー生成コンテンツ(UGC)**による自然流入増加
- **コミュニティ内での求人拡散**（紹介経由の応募増）
- **退会されない仕組み**（転職後も継続利用）

##### 2. AI/MLによるマッチング精度向上

**現状の課題**: 人力マッチングの限界
- キャリアアドバイザーの処理能力: 1人あたり月間30-50件
- マッチング精度: 主観・経験に依存

**機械学習導入後**:
```
学習データ:
├── 過去の応募データ
├── 内定・辞退データ
├── 診断結果×求人のペア
└── ユーザー行動ログ

予測タスク:
├── 応募確率予測（レコメンデーション最適化）
├── 内定確率予測（応募者への期待値管理）
└── 早期退職リスク予測（ミスマッチ防止）
```

**ビジネスインパクト**:
- **応募数**: 適合度の高い求人を優先表示 → 応募率1.5倍
- **内定率**: 通過確率の高い企業を推薦 → 内定率1.3倍
- **定着率**: ミスマッチ防止 → 早期退職率30%減

**技術投資**: データサイエンティスト1名+インフラ → 年間コスト1,500万円  
**効果**: マッチング効率化で売上10-20%増 → 投資回収1-2年

**技術的アプローチ**:
```

    └── 税理士向けロジックがハードコード
```

##### 5. セキュリティ・プライバシー投資の戦略的価値

**重要な視点**:  
データが資産なら、**セキュリティ投資は資産保全コスト**である。

**投資対象**:

| 項目 | 投資内容 | 予算 | 効果 |
|-----|---------|------|------|
| **データ暗号化** | DB暗号化、通信TLS1.3化 | 200万円 | 漏洩時の被害最小化 |
| **アクセス制御** | RBAC、監査ログ | 300万円 | 内部不正防止 |
| **脆弱性診断** | 年2回のペネトレーションテスト | 100万円/年 | 事前のリスク発見 |
| **GDPR/個人情報保護** | 同意管理、削除権対応 | 500万円 | 法令遵守、信頼獲得 |

**ビジネス価値**:
- **Pマーク・ISMSの取得**: 大手企業からの受注が可能に
- **上場準備**: 監査に耐えうる体制構築
- **データ販売**: コンプライアンス体制が前提条件

**ROI**: セキュリティ投資1,000万円 → 企業信頼度向上 → 大口契約獲得（年間3,000万円）

##### 6. 技術的優位性による競合参入障壁

**現状の課題**: 求人サイトは参入障壁が低い（WordPress + プラグインで構築可能）

**技術投資による差別化**:

```
Level 1（参入容易）: 求人サイト
└── WordPress + 求人プラグイン
    └── 開発コスト: 100万円、期間: 1ヶ月

Level 2（やや困難）: 診断機能付き求人サイト
└── Laravel + カスタム診断
    └── 開発コスト: 500万円、期間: 3ヶ月

Level 3（困難）: データプラットフォーム
└── API基盤 + 分析環境 + ML
    └── 開発コスト: 3,000万円、期間: 12ヶ月
    └── ★ミツプロの目指す場所

Level 4（極めて困難）: 業界OS
└── マルチテナント + リアルタイム + コミュニティ
    └── 開発コスト: 1億円+、期間: 24ヶ月+
```

**戦略的示唆**:  
Level 3-4に到達すれば、**後発企業が追いつくのに2-3年かかる**。  
その間に、ネットワーク効果（ユーザーが増えるほど価値が高まる）で市場を独占できる。

#### C. 統合的な価値創出シナリオ

マーケッターとCTOが**協働**することで、最大の価値を生み出せます。

**フライホイール効果**:
```
① SEOコンテンツ（マーケ）
   ↓
② ユーザー流入増加
   ↓
③ 診断データ蓄積（CTO: データ基盤）
   ↓
④ より精緻な市場分析が可能に
   ↓
⑤ さらに質の高いコンテンツ生成（マーケ）
   ↓
① に戻る（好循環）
```

**具体的な協働例**:

| マーケッターのニーズ | CTOの提供価値 |
|---------------------|--------------|
| 「診断からの離脱が多い」 | A/Bテスト基盤を構築、データドリブンな改善 |
| 「競合との差別化が難しい」 | 独自データを活用した市場レポート生成ツール |
| 「リターゲティング精度を上げたい」 | セグメント管理API、外部広告ツールとの連携 |
| 「コミュニティを作りたい」 | リアルタイムチャット、掲示板機能の実装 |

**組織としての競争力**:  
データ × 技術 × マーケティングの三位一体で、**単なる求人サイトを超えた業界インフラ**へ。

## 2. 現状分析とリスク評価

### 2.1 技術的負債の整理

| カテゴリ | 内容 | ビジネス影響 | 対応優先度 |
|----------|------|--------------|------------|
| **セキュリティ** | 認証情報の漏洩履歴、EOLフレームワーク | 不正アクセス、コンプライアンス違反 | 最優先 |
| **基盤** | Laravel 8 EOL（2023年1月） | 脆弱性対応不可、新規パッケージ利用困難 | 高 |
| **フロントエンド** | Tailwind/Bootstrap/Alpine/Vue混在 | 開発速度低下、バグ増加リスク | 中 |
| **テスト** | カバレッジ不足（一部機能にテストなし） | リグレッション、品質不安定 | 中 |
| **コード品質** | 命名規則不統一、デバッグコード残存 | 保守コスト増加 | 低 |

### 2.2 リスクマトリクス（役員向け）

```
影響度
  ↑
高 │ 【セキュリティ】認証情報漏洩  【基盤】Laravel EOL
  │     → 早急に対応が必要
  │
中 │ 【フロント】技術混在  【テスト】カバレッジ不足
  │     → 計画的に対応
  │
低 │ 【コード品質】命名・デバッグコード
  │     → 余裕があれば対応
  └────────────────────────────→ 発生確率
      低        中        高
```

### 2.3 強み（現状維持・活用）

- **Docker環境**: 開発環境の再現性が高く、オンボーディングが容易
- **CI/CD**: GitHub Actionsでテスト自動実行が稼働中
- **ドキュメント**: 開発ガイド、技術負債ロードマップが整備済み
- **テスト基盤**: 84テストケースが存在し、主要機能はカバー済み

## 3. 運営方針の基本原則

### 3.1 開発の優先順位ルール

1. **セキュリティ・安定性** > 新機能開発
2. **技術負債の返済** と **新機能開発** を **並行** で進める（負債のみに偏らない）
3. **テストを書いてから** 大規模変更を行う（リグレッション防止）
4. **段階的リリース** を原則とし、一括大型リリースを避ける

### 3.2 意思決定の基準

| 判断が必要な場面 | 基準 |
|------------------|------|
| 新機能の優先度 | ビジネス価値 × 技術的リスク で評価。基盤が不安定な場合は基盤を優先 |
| 技術選定 | 既存スタックとの整合性、学習コスト、コミュニティの活発さを考慮 |
| スケジュール調整 | 品質を犠牲にした納期短縮は行わない。スコープ調整で対応 |
| 外部委託 | ドキュメント整備済みの領域から委託。知識の属人化を防ぐ |

### 3.3 品質ゲート（リリース前チェック）

- [ ] 全テストが成功
- [ ] セキュリティスキャン（依存パッケージの脆弱性チェック）実施
- [ ] 本番環境と同等の環境での動作確認
- [ ] ロールバック手順の文書化

## 5. 新機能開発のプロセス

### 5.1 開発フロー

```
【要件定義】→【設計レビュー】→【実装】→【テスト】→【コードレビュー】→【リリース】
     ↑              ↑                ↑
  ビジネス側     正社員エンジニア    品質ゲート
  との合意       の技術レビュー      必須
```

### 5.2 新機能の技術審査チェックリスト

新機能を着手する前に、以下を確認します：

| 項目 | 確認内容 |
|------|----------|
| 既存機能との整合性 | 同じパターン・規約で実装できるか |
| テスト方針 | 単体・結合・E2Eのどこまでカバーするか |
| パフォーマンス | 想定ユーザー数・データ量で問題ないか |
| セキュリティ | 認証・認可・入力検証は適切か |
| フロントエンド | 統一方針（Tailwind + Alpine）に準拠するか |

### 5.3 禁止事項（技術的デットライン）

- 本番環境の認証情報をコードやコミットに含める
- テストなしで本番にリリースする
- 技術負債ロードマップに反する技術選定（例：新規でBootstrapを追加する）
- ドキュメントを更新せずにアーキテクチャを変更する

## 6. メンテナンス・運用方針

### 6.1 定期実施項目

| 頻度 | 項目 | 担当 |
|------|------|------|
| 毎日 | 監視アラート確認、ログ確認 | 運用担当 |
| 毎週 | 依存パッケージの脆弱性チェック（`composer audit`） | 開発担当 |
| 毎月 | バックアップのリストアテスト、パフォーマンス確認 | 開発・運用 |
| 四半期 | 技術負債ロードマップの見直し、優先度の再評価 | 開発チーム全体 |

### 6.2 障害対応フロー

1. **検知**: 監視・ユーザー報告・ログ確認
2. **初動**: 影響範囲の特定、ステークホルダーへの連絡
3. **対応**: 原因調査、暫定対応（ロールバック等）、恒久対応
4. **事後**: ポストモーテム作成、再発防止策の実施

### 6.3 バックアップ・復旧

- データベース: 日次バックアップ（推奨）
- 設定・コード: Gitで管理、タグでリリース版を管理
- 復旧目標: RTO（目標復旧時間）48時間以内、RPO（目標復旧時点）24時間以内を目安に設計

## 8. 予算・コストの考え方（役員向け）

### 8.1 技術投資の分類

| 分類 | 例 | 性質 |
|------|-----|------|
| **必須投資** | セキュリティ対応、Laravelアップグレード | やらないとリスクが増大。優先して実施 |
| **効率化投資** | フロントエンド統一、テスト強化 | 中長期で開発コストを削減。計画的に実施 |
| **成長投資** | 新機能、API拡張 | ビジネス成長に直結。バランスよく実施 |

### 8.2 コスト削減時の優先度（逆順）

削減が必要な場合、**最後に削る**べきもの：

1. セキュリティ対応
2. バックアップ・監視
3. Laravelアップグレード（EOL対応）
4. テスト・CI/CD

**先に検討できる**もの：

- フロントエンド統一のスケジュール延長
- 新機能のスコープ縮小
- 一部のコード品質改善の先送り

## 11. 付録：用語集（役員向け）
